Three Approaches for Live Text Editor

Each approach differs in architecture and complexity. Click to select your preference.

A

Standalone Vite Plugin (Recommended)

A single vite-plugin-live-edit package that bundles everything:

  • Babel transform — injects data-source-file / data-source-line into JSX text elements at compile time
  • Overlay runtime — a React component auto-injected in dev mode that handles the floating popover, annotation mode, keyboard shortcuts, and floating button
  • Dev server middleware — an API route on the Vite dev server (/__live-edit/save, /__live-edit/annotations) that writes changes back to source files and manages .annotations.json

Pros

  • One line to add: liveEdit() in vite.config
  • Zero app code changes needed
  • Portable to any Vite + React project
  • Clean separation from app code

Cons

  • Babel transform adds ~50-100ms to dev startup
  • More upfront architecture work
B

Two-Part: Plugin + React Provider

Split into a Vite plugin (Babel transform + server middleware) and a React context provider you wrap your app with:

  • Vite plugin — handles the Babel transform and server-side file writing
  • React provider — <LiveEditProvider> wraps your app, provides edit/annotate mode context, renders the popover and floating button
  • You manually add <LiveEditProvider> in your App.jsx (dev only)

Pros

  • More control over overlay behavior per-project
  • Can pass config props to the provider
  • Easier to debug (visible in React DevTools)

Cons

  • Requires app code change (wrapping with provider)
  • Two integration points instead of one
  • Provider must be removed/guarded for production
C

Bookmarklet + WebSocket Server

A completely decoupled approach — no build-time changes at all:

  • Bookmarklet / browser extension — injects the overlay UI into any page
  • Separate Node server — runs alongside Vite, connected via WebSocket, handles file search + write-back
  • Source mapping done at runtime by text-matching visible content against source files (fuzzy search)

Pros

  • Zero build config changes
  • Works with any framework, not just React
  • Could work on deployed sites too

Cons

  • Fuzzy text matching is unreliable (duplicates, dynamic text)
  • No exact line numbers — best-effort source location
  • Separate server to manage
  • Bookmarklet UX is clunky

My Recommendation: A

The standalone Vite plugin is the cleanest approach. One config line, zero app code changes, and the Babel transform gives you exact source locations — which is the foundation everything else depends on. The slight build-time cost is negligible in dev mode. Option B adds unnecessary integration friction for no real gain. Option C sacrifices the most important feature (reliable source mapping) for flexibility you don't need.
